- 本合集是"动手前 / 下结论前 / 重构时 / 沟通时"所有通用纪律的单一来源。

## § 1. 元规则：调研与输出的根基（优先级：最高）

> 本节是所有涉及代码事实输出的元规则，无关键词依赖、无场景豁免。任何下层条款与本节冲突时以本节为准。

### 1.1 调研完整性

任何**方案、结论、判断、观点**，输出前，必须自行完成**证据**确认：即确认整个结论赖以成立的、不可降级的关键代码事实，并且能够提供对应的代码或文档证据（原文 + 行号）。

- 基于类名 / 变量名 / 经验类比的"应该是 / 通常是 / 大概率是"——这些不是**证据**，是**假设**。假设**禁止**进入给用户的回答。
- 在声称"代码 X 是错的，原因是 Y"之前，必须真正读过 X 的实现（它调用的函数、相关函数的语义、它演化到今天的历史）。绝不能仅凭标识符字面含义构造批评。无法引用到所批判的具体「函数名 + 行号」 = 调研不足 = 不足以形成观点 = 不下结论。

无法提供**证据**的**方案、结论、判断、观点**，只能三选一处理：

- **(a) 变成工具调用**：去 read_file / grep_search 读到原文行号后再写；
- **(b) 变成疑问抛给用户**：仅限代码库外的信息（产品意图 / 外部契约 / 用户偏好）；**绝对严禁**不充分调研就把疑问抛给用户，**绝对严禁**提问代码库内可以查到的信息；
- **(c) 从答复中删除**：若该陈述对当前问题非必要，直接不写。

### 1.2 调研自主性

**禁止把调研外包给用户**，所有对代码库内部的只读类查询 / 调研，都不需要也不应该征求用户的同意。凡是通过 read_file / grep_search / view_code_item 等**工具调用**就能确认的代码库内事实，必须必须在输出**方案、结论、判断、观点**之前由 Agent 自己主动调用工具查清，不许留在任何形式的待确认事项中。

- **"待确认事项" / "不确定项"列表**只允许包含代码库外的产品意图 / 用户偏好 / 外部契约问题，**不允许**包含任何 read_file / grep_search 能解决的代码事实。

- **禁止**以下"礼貌性请示"式的懒惰行为：

  - "请您确认是否允许我去读 XXX"

  - "是否同意我先看一下 YYY"

  - "要不要我去查 ZZZ"


用户只应为"代码库外信息"做决策（产品意图 / 用户偏好 / 命名偏好 / 外部契约 / 方案选型）。代码库内的事实确认是 Agent 的本职工作。

违反本条 = 把本该自己做的工作外包给用户 = **严重违规**。

## § 2. 行动方法论：编辑与重构

### 2.1 编辑前自检（强制流程）

每次编辑代码前，必须明确列出三件事：

- **(a)** 要改哪个函数（精确到文件 + 函数名 / 字段）；
- **(b)** 改什么内容（新增 / 删除 / 替换哪几行，为什么）；
- **(c)** 改完后各相关函数的职责是什么（有没有职责迁移 / 是否破坏现有分工）。

然后在真正发起编辑工具调用之前，必须再做一次验证：编辑工具的 `old_string / new_string / target` 是否与上面的分析结论一致。

"分析说改 A，实际却改了 B" = 严重违规。

### 2.2 不基于臆测的需求写代码（No-fabricated-requirements）

禁止为"可能的未来需求"提前写分支 / 抽象 / 扩展点。

超出当前明确需求的逻辑，二选一：

- **(a)** 引用具体的现有调用点 / 代码证据，证明该场景在当前代码库中已经存在；
- **(b)** 显式向用户确认该未来需求，并等待批准。

只实现已知需求的最小路径。发现潜在扩展点时，单独列出告知用户，**不要**静默实现。

### 2.3 大型重构 / 架构变更中的冲突上报

执行一个已获用户批准的大型重构或架构改动时，如果在实施过程中发现"批准的方案"与"代码实际情况"存在冲突（例如隐藏的复杂度、逐帧状态依赖、生命周期不匹配等），必须 **STOP**，向用户清晰陈述冲突分析和可选决策路径，而不是默默偏离原计划继续推进。

## § 3. 行动方法论：代码风格与规范

### 3.1 下游信任

信任下游逻辑自然过滤，即caller应尽量信任callee内部逻辑，不要在caller层为callee做判空等提前检查。caller层仅作本层应做的检查，不为callee层做保证

### 3.2 命名直接反映真实语义

变量命名语义直接，优先反映真实含义；不为追求简短而牺牲语义。

### 3.3 注释只描述当前语义，不描述演化历史

**核心原则**：注释只描述代码当前版本的语义，**不描述演化历史**。新读者打开代码时，注释应该让他理解"这段代码现在在做什么、为什么这样做"，而不是"这段代码以前是什么、为什么改成现在这样"。

注释严禁出现以下内容：
1. **迭代信息 / 决策史**：禁止出现"以前是 X 现在改为 Y"、"重构后 / 同意的语义"、"被打回过 / 这次改成"、"Q1/Q2-A/Q4 = ① 同意 / 接受"等任何指向"代码演化过程"的字样。这些信息属于提交信息 / PR 描述，不属于源码注释。
2. **与对照实现的横向对比解释**：禁止"与 X 不同，因为 Y 的特殊性所以这里 Z"这种通过对比来解释的写法——它假设读者也在做对比阅读，而新读者不会。直接陈述当前代码的语义，对比信息留给读 PR 的人。
3. **重复字段名 / 函数名已表达的含义**：字段名叫 `FadeInDuration` 就不要再写 "/** 渐入持续时长 */"。注释只在添加价值时存在。
4. **形参伪注释**（如 `/*InXxx=*/`、`/*ElapsedDuration=*/`、`/*bAllowFadeOut=*/`）：IDE 自带形参提示，伪注释纯粹冗余。
5. **Agent 内部推理草稿**（如"重推：……等一下，不对……问题继续推导……"）：这是 Agent 的思考草稿，禁止留在源码里。

### 3.4 C++声明与实现

.cpp 文件中的函数实现顺序必须与对应 .h 文件中的声明顺序保持一致。
